Tech@NYU API Documentation

This API lets clients query and manage almost all of tech@nyu’s organizational data. For example:

  • The Ship archive uses it to retrieve all the Projects we’ve shipped, while the Demo Days archive uses it to retrieve all the Projects shown at a given Demo Days, as well as information about who spoke and where the event was.
  • The primary tech@nyu website uses its Events to power the event calendar, and to show the current board members and their Team affiliations.
  • An eboard member might query the API in search of a speaker or venue for his/her event, as the API can list Presenters and Venues from prior tech@nyu events. For presenters, it also shows the expertise of each and the tech@nyu events they’ve presented at.

Usage Overview

This API is compliant with the JSON API specification, which defines conventions for listing, reading, creating, updating, and deleting resources. When fetching a resource or resources, the spec also offers conventions for: sorting the returned resources, limiting the fields returned in each resource, including related resources to form compound documents, etc. Accordingly, these features of the Tech@NYU API aren’t redocumented here; simply see the JSON API spec to learn how to use them.

Unsupported JSON API features

This implementation of JSON API does not allow clients to set a resource’s id; any attempt to do so will be silently ignored, with the server-generated id being used instead. This implementation also does not yet support PATCH operations of any kind or using relationship endpoints.

Authorization

Currently, only one authorization scheme has been implemented, and it’s designed to support browser-based JS apps that need access access to protected resources based on a user’s provided credentials. The user verifies himself by signing in with Facebook, and, upon doing so, he receives certain privileges through roles associated with his account.

But because browser-based apps can’t keep secrets (i.e. they’re code is completely transparent), they shouldn’t directly initiate the OAuth handshake with Facebook. So instead, the scheme works as described below.

Signing In

  1. The JS app presents a button to the user that, when clicked sends the user to https://api.tnyu.org/v2/auth/facebook?success={goodUrl}&error={errUrl}, where goodUrl and errUrl are the urls for the pages in the application that the user should be sent to when, respectively, the login is complete or the login fails. The error page isn’t given any information about the source of the error, though this may change. For now, it should simply promp the user to log out and try signing in again. To just login to the API https://api.tnyu.org/v2/auth/facebook
  2. Assuming authorization succeeded, the server will set a standard session cookie associated with api.tnyu.org, which will be used by the server to allow access to protected resources on subsequent requests. All traffic to api.tnyu.org is encrypted with SSL, to prevent session hijacking etc.

Logging Out

Simply send a GET request to https://api.tnyu.org/auth/facebook/logout.
